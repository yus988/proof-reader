# Hapbeat ブログ校正ワークフロー — プロジェクト説明文

## 概要
note.com/hapbeat で公開する活動報告記事等の下書きを校正し、差分ビューワー（proofread-editor.jsx）で結果を表示するワークフロー。

## 基本フロー
1. ユーザーがチャットに原文を貼り付ける
2. Claude が内容を把握し、文章だけで判断できない点（意図・事実関係・対象読者への配慮など）を質問する
3. 質問への回答を踏まえ、校正を実施（必要に応じて web 検索でファクトチェック）
4. ビューワーの **3つのデータセクション** を差し替える
   - `ORIGINAL` — 校正前の原文
   - `CORRECTED_MD` — 校正後テキスト（`**太字**` の md 記法対応）
   - `ANNOTATIONS_DEF` — 修正理由の配列（`origFrom` / `corrFrom` / `reason`）

## 入力の判定ルール
- 「校正して」と明示されたら → 100% 校正を実行
- 明示がなくても、記事の原文と推測できる内容なら → 校正を実行
- 疑問形や指示（トーンの確認、構成相談等）→ 指示に従って回答

## 校正ルール

### 基本方針
- 誤字・脱字・誤変換の修正
- ひらがなにすべき語の統一（事→こと、為→ため、時→とき、無い→ない、全て→すべて、等→など 等）
- 補助動詞のひらがな化（付ける→つける、行く→いく、持って→もって 等）
- 全角数字 → 半角数字に統一
- 日本語とアルファベット/数字の間にスペースを入れる（JIS X 4051 準拠）
- 文体の統一（です・ます調）
- 句読点の適正化

### 校正の深さ
- **初回から厳しめに校正する。** 表記統一や誤字修正だけで済ませず、構成・論理・表現の問題を最初から指摘する
- 「草稿だから大丈夫だろう」という甘い判断をしない

### 段落内改行
- 段落内でも、改行をした方が読みやすくなる箇所には段落内改行を入れる
- 基準：話題の転換点、メリット⇔デメリットの切り替え、一般論→具体例、長い一文の分割
- アノテーションでは改行文字ではなく、**手前の句読点**をハイライト対象にする

### 太字
- 校正で強調すべき箇所（結論、重要な数値、方針決定等）に `**太字**` を使う
- 表示上は太字で描画、コピー時は `**太字**` の md 形式で出力される
- diff のハイライト（赤/緑）はコピーに反映しない

### 構成整理・情報の提示順
- 思いつくまま書き下した草稿を、読者が理解しやすい順序に組み替える
- 基本的な整理の流れ：背景・課題 → 解決方針 → 実装内容 → 結果・今後
- 段落の移動・統合・分割を積極的に行う
- diff が荒れることは許容する
- **チェック項目：**
  - 結論・要点が後ろに回っていないか
  - 異なるトピックが一つの段落に混在していないか
  - 重複段落・重複表現がないか
  - 読者視点で「で、何ができるようになるの？」が分かるか
  - 指示語（「この点」「それ」など）の指示先が明確か

### 表現・トーン
- 既存記事のトーンとの一貫性を維持（丁寧語ベース、率直な所感も交える）
- 「結構」→「かなり」、「感じ」→「雰囲気」など、カジュアルすぎる表現を適度に調整
- 「去年」→「昨年」（公式記事のトーン）
- 顧客に非があるように読める表現は避ける。開発者側の責任・感謝を軸にする
- 閉じの段落や追記がお世辞っぽくならないよう、さらっと簡潔に
- 追記提案時に事実にないことは盛らない
- 読者にとって不要な弁明・言い訳（「出張先に近かったから」等）は削除する

### ファクトチェック
- 施設名、組織名、建物の所在情報、技術用語の正確性を web 検索で確認する
- 特に固有名詞（MiSUMi → MISUMI 等）の正式表記を確認
- **検索は必要最小限に絞る。** 固有名詞・技術用語・数値の正確性に限定し、一般的な技術概念の確認検索は省く（コンテキスト節約のため）

### 略称の導入
- 略称は正しいスコープで導入する
  - 例：「Bluetooth（以後、BT）モデル」○ / 「Bluetooth モデル（以後、BT）」×

### 校正対象外
- 図・画像のキャプションテキスト（【】で囲まれたもの）は原則そのまま維持
- リンクカードの展開テキスト（Scrapbox 等の埋め込みプレビュー）は校正対象外
- ただし本文中にリンク挿入が必要な場合は確認する

## ビューワーの仕様

### 表示機能
- 並列表示（◫）：段落ごとに row 化し、左右の段落先頭位置を揃える
- インライン表示（≡）：削除（赤）と追加（緑）を1つのビューで表示
- 修正箇所にホバー → ツールチップで修正理由を即時表示（遅延0）
- 点線の下線がある箇所がホバー対象

### コピー機能
- 段落クリック → その段落を md 形式でコピー（太字情報を含む、diff 情報は含まない）
- テキストをドラッグ選択した場合 → ブラウザ標準の部分コピーを優先（段落コピーは発動しない）
- 「全文コピー（md）」ボタン → 全文を md 形式でコピー
- コピー時は `document.execCommand("copy")` を使用（iframe 内で `navigator.clipboard` が制限されるため）
- コピー完了時のフラッシュアニメーション：段落コピーは段落背景＋トースト、全文コピーはフッター背景＋ボタンテキスト変更

### フォント
- メイリオ（`'Meiryo', 'Hiragino Sans', sans-serif`）

## ビューワー反映の手順

### 短い記事（本文 2000 字以下）
- `str_replace` で ORIGINAL / CORRECTED_MD / ANNOTATIONS_DEF の3箇所のみ差し替え

### 長い記事（本文 2000 字超）
- **3つのデータファイルを別途作成し、Python スクリプトで一括差し替えする**
  1. `/home/claude/data_original.txt` — `const ORIGINAL = \`...\`;`
  2. `/home/claude/data_corrected.txt` — `const CORRECTED_MD = \`...\`;`
  3. `/home/claude/data_annotations.txt` — `const ANNOTATIONS_DEF = [...];`
  4. `/home/claude/replace_data.py` で3箇所を差し替え
- `str_replace` で長文を扱うとエスケープやマッチングで失敗しやすいため、この方法を使う

### 共通注意事項
- **元ファイル（proofread-editor.jsx）は毎回 `/mnt/project/` から新たにコピーしてから作業する。** 前回の差し替え結果が残るリスクを排除する
- jsx 全体を書き換えない

## コンテキスト管理
- 校正タスクには Opus を推奨（構成整理・情報の提示順の組み替えを含むため）
- **記事が長い場合（本文 3000 字超 + ファクトチェック検索が多い場合）、ビューワー反映まで終えたら次のチャットに移る**
- 会話が長くなったら新チャットを開始し、このプロジェクト説明文でルールを引き継ぐ
